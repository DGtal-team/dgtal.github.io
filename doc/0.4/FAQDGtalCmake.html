Introduction and first build

The DGTal library uses cmake to manage the project build on various systems. With the same project description syntax, cmake can do various things:

  • Check the compiler and its options
  • Check and locate the project dependencies
  • Generate different build backends (Unix makefile, VisualStudio projects,..), also called generators.
  • Create documentation build rules
  • Manage unit tests
  • ...

Build on Linux-based distributions

An important feature of cmake is the "out-of-source" build process. Let us suppose that DGtal sources are in the DGTALSOURCES folder. Hence, in a terminal you can create a build folder: 

cd DGTALSOURCES
mkdir build
cd build

Next, use the cmake command to generate your Unix Makefile project (default generator on unix/linux): 

cmake ..

If no error occurs, you can build the DGtal lib and install it: 

make 
sudo make install

Build on Mac Os

Since recent MacOs distribution are unix-like operating systems. The easiest way to compile DGtal is to follow Unix/Linux instructions.

Build on Windows/Visual Studio

Use the Cmake Windows interface to generate your Visual Studio project. Again, a good practise is to specify a different folder in which cmake and VS will put their build results.

On windows, there is no canonical locations on the folder tree where cmake could find the include files and the DLL of the DGtal dependencies. Hence, you will probably have to manually set the paths to find the boost headers and DLL for example.

Once the VS project as been created, you just have to simply open it with VS.

DGtal cmake options

In DGtal CmakeLists, serval options can be specified to custom the library build. First, cmake has predefined build options (CMAKE_BUILD_TYPE) such as:

  • CMAKE_BUILD_TYPE
    • Debug: debug mode (e.g. C++ debug flags, DEBUG variable, ...)
    • Release: release build (optimized C++ compiler options, NDEBUG variable,...)
  • CMAKE_INSTALL_PREFIX: a string containing the DGtal install path (e.g. by default on some linux: "/usr/local/").

Beside these options, user can also specify:

  • BUILD_EXAMPLES (ON/OFF): to build DGtal examples (default: ON)
  • BUILD_TESTING (ON/OFF): to build the test suite (default: ON)
  • BUILD_TOOLS (ON/OFF): to build DGtal tools (default: ON)
  • BUILD_INLINE_FUNCTIONS (ON/OFF): to build inline functions (default: ON)
  • DEBUG_VERBOSE (ON/OFF): to define the DEBUG_VERBOSE variable (default: OFF). If set to ON and in Debug build type, verbose debug messages are traced out.
  • BUILD_SHARED_LIBS (ON/OFF): to build the lib as a shared library (default: ON)

HowTos

Add a new class in DGtal

To add a new class on linux, we encourage you to use the dedicated script in the DGTALSOURCES/dev/scripts folder. More precisely:

  • Create a new profile in DGTALSOURCES/dev/profiles (check existing profile files to see the syntax and the name of the brand new file should match with you system login
  • Then, the DGTALSOURCES/dev/scripts/newclass.sh script allows you to create a new class from several models. E.g. 
    newclass.sh MyNewDomainClass kernel/domains
    will create three files in the DGTALSOURCES/src/DGtal/kernel/domains folder:
    • MyNewDomainClass.h
    • MyNewDomainClass.ih for inlined functions
    • MyNewDomainClass.cpp for definitions of methods.

If you intend to create a class with template arguments (usually the case in DGtal), use the newtemplateclass.sh script instead (it will only generate .h and .ih files).

You can now edit the files and add them to the build system.

If your class contains a implementation file (i.e. a CPP file), it must be added to the list of prerequisites for the main DGtal lib target. Hence, let us suppose that you would like to insert DGTALSOURCES/src/DGtal/kernel/domains/MyNewDomainClass.* in the build system. Hence, you have to edit the corresponding DGTALSOURCES/src/DGtal/kernel/ModuleSRC.txt file in order to add a new line with DGtal/kernel/domains/MyNewDomainClass. Indeed Each DGtal module (kernel module in this example) has got a ModuleSRC.txt file containing the list of classes to add to the DGtal lib target.

If your class is a templated class, you don't have to edit the ModuleSRC.txt file.

Add a new test file class

In DGtal, we use a test suite in order to check the consistence of the library through the time. When you add a new class, it's very important to add one or more tests files in the folder 

DGTALSOURCES/tests/MODULE/

Basically, a test file is an executable with a return code 0 if all the tests of the file are satisfied and 1 otherwise. Unit testing is a method by which individual units of source code are tested to determine if they are fit for use. If they are, the return code 0 is used. In the dev folder, there is script and a file skeleton that help you creating a test file for a given functionality. In that case, you can simply

newtest.sh MyNewDomainClass kernel/

This script will add a testMyDomainClass.cpp in the DGTALSOURES/tests/kernel folder. You can now edit this file to create your test. The last step is to add it to the build system. To do that, just edit the DGTALSOURES/tests/kernel/CMakeLists.txt file and add a new line with testMyDomainClass in the test suite.

To run all the DGtal tests, you can either build the tests target on VS or to a make test on linux.

Add a new example file in DGtal

To add a new example (clean and educational piece of code), the process is similar to the previous section with the script 

newexample.sh simpleExample kernel/

Add a new dependency in DGtal

To add a new dependency (e.g. ExtraLib in the following), several things have to be done:

  1. Create an appropriate FindExtraLib.cmake script in the DGTALSOURCES/cmake/ folder;
  2. Edit the file DGTALSOURCES/cmake/CheckDGtalDependencies.cmake and cute/paste a new block starting with the FIND cmake command. If cmake detects the library, it is important to append such lib to the DGtalLibDependencies variable.
  1. If the dependency is detected, a C definition "USE_<DEP>" must be defined and transfered to the compiler (e.g. "USE_ITK")

Add a new documentation file

For each DGtal feature or module, specific documentations are required (user-guide and manual). The documentation file must be a valid doxygen file, added in to the module folder (e.g. src/DGtal/kernel/domains/newDomain.dox). To easily create such file, you can use the DGTALSOURCES}/dev/scripts/newdocumentation.sh script. Once you're done, edit the file DGTALSOURCES/src/DGtal/base/Config.h.in and add a \ref to the new section added in the new file newdomain.dox.
